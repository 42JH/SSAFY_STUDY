# 도입 : Clean Code, Technical Writing은 무엇이며, 왜 필요한가?

**Clean Code**란?

단순히 동작하는 코드가 아니라

> **다른 개발자가 읽고 이해하기 쉬운 코드**를 의미한다.
> 

대표적인 특징은 다음과 같다.

- 의미 있는 네이밍
- 단순한 함수 구조
- 명확한 로직

즉, 코드를 작성한 사람이 아니라

**처음 보는 개발자도 이해할 수 있는 코드**가 Clean Code이다.

반면 **Technical Writing**은

> **코드와 시스템을 문서로 설명하는 기술적인 글쓰기**를 의미한다.
> 

대표적인 예는 다음과 같다.

- README
- API 문서
- 시스템 아키텍처 문서
- 개발 가이드

정리하면

- **Clean Code → 코드 내부의 커뮤니케이션**
- **Technical Writing → 코드 외부의 커뮤니케이션**

---

그러면 이 두 가지가 필요한 이유는?

소프트웨어 개발에서 가장 많은 비용이 들어가는 단계는

**새로운 기능 개발이 아니라 유지보수(Maintenance)** 이다.

여러 연구에 따르면 실제 소프트웨어 개발 시간의 **60~80%는 기존 코드를 이해하는 데 사용된다.**

문제는 대부분의 코드가 **작성자의 머릿속에만 설계가 존재한다는 것**이다.

- 코드에 의도가 드러나지 않는다
- 문서가 부족하다
- 다른 개발자가 이해하기 어렵다

이 경우 팀원들은 결국 다음과 같은 질문을 반복하게 된다.

> "이 코드 왜 이렇게 작성된 거죠?"
> 

이런 상황은 단순한 커뮤니케이션 문제가 아니라

**개발 생산성의 병목(Bottleneck)** 이 된다.

그래서 등장한 개념이 **Clean Code와 Technical Writing**이다.

# 본론 1 : 네이밍 컨벤션 - 이름만으로 의도 전달하기

코드를 읽을 때 개발자가 가장 먼저 보는 것은 **변수 이름과 함수 이름**이다.

좋은 네이밍은 코드 이해 시간을 크게 줄여준다.

---

## 1️⃣ 변수명에 의미와 단위 포함

나쁜 예시 : 의미와 단위가 명확하지 않은 변수명

```
constdelay=10  # 무엇을 의미하는 값인지 알기 어려움
```

이 값이 의미하는 것은 무엇일까?

- 10초?
- 10밀리초?
- 10프레임?

좋은 예시 : 단위를 포함하여 의미를 명확히 표현한 변수명

```
constdelay_ms=10  # 밀리초 단위를 명확히 표현
```

단위를 포함하면 코드 해석이 훨씬 쉬워진다.

이 방식은 특히 다음 분야에서 중요하다.

- 데이터 분석
- 머신러닝
- 성능 최적화

예

```
timeout_seconds  # 요청 대기 시간 (초)
batch_size       # 모델 학습 시 한 번에 처리하는 데이터 크기
learning_rate    # 모델 학습 속도를 조절하는 파라미터
```

---

## 2️⃣ 함수명은 동사 중심으로 작성

함수 이름은 **행동(Action)을 설명해야 한다.**

예시 :

데이터 / AI

```
def preprocess_data(data):   # 데이터 전처리를 수행
    ...

def train_model(dataset):    # 모델 학습을 수행
    ...

def evaluate_model(model):   # 모델 성능 평가
    ...
```

PM / 비즈니스 로직

```
def validate_requirements(req):   # 요구사항이 유효한지 검증
    ...

def calculate_roi(cost,profit):   # 투자 대비 수익 계산
    ...

def generate_report(data):        # 분석 결과 보고서 생성
    ...
```

함수 이름만 보고도 **이 함수가 무엇을 하는지 추측할 수 있어야 한다.**

---

## 3️⃣ Boolean 변수 네이밍 규칙

흔히 Flag 변수라고도 불리는 Boolean 변수는 조건문 가독성을 높이기 위해 보통 다음 접두어를 사용한다.

- is
- has
- should
- can

예시 : is / has / should를 사용하여 변수를 네이밍

```
is_valid_user = True      # 유효한 사용자 여부
has_permission = False    # 접근 권한 보유 여부
should_retry = True       # 재시도 필요 여부
```

이 규칙은 조건문 가독성을 크게 개선한다.

# 본론 2: Clean Code 실무 전략 - 함수 구조와 주석

좋은 코드는 단순히 네이밍만으로 만들어지지 않으며

**함수 구조와 주석 작성 방식** 역시 중요하다.

---

## 1️⃣ 함수는 하나의 책임만 가진다

소프트웨어 설계에서 가장 중요한 원칙 중 하나는

**Single Responsibility Principle(SRP)** 이다.

> **하나의 함수 또는 클래스는 하나의 책임(Responsibility)만 가져야 한다.**
> 

여기서 말하는 책임(Responsibility)란, **코드가 수행하는 하나의 기능 또는 역할**을 의미한다.

나쁜 예시 : 여러 책임이 하나의 함수에 포함되어 있음

```
def process_user_data():
	users=load_users()               # 사용자 데이터 로드
	valid_users=filter_users(users)  # 유효한 사용자만 필터링
	save_users(valid_users)          # 결과를 데이터베이스에 저장
```

이 함수는

- 데이터 로드
- 데이터 필터링
- 데이터 저장

세 가지 일을 동시에 수행한다.

개선한 버전 : 책임을 분리하여 함수 구조 개선

```
def load_users():
    ...                 # 사용자 데이터를 데이터베이스에서 불러온다

def filter_valid_users(users):
    ...                 # 유효한 사용자만 필터링한다

def save_users(users):
    ...                 # 처리된 사용자 데이터를 저장한다
```

이렇게 분리하면

- 테스트가 쉬워지고
- 유지보수가 쉬워지고
- 재사용성이 높아진다.

### 참고 : SOLID 설계 원칙

SRP는 객체지향 설계 원칙인 **SOLID 원칙** 중 하나이기도 하다.

SOLID는 **유지보수성과 확장성을 높이기 위한 다섯 가지 소프트웨어 설계 원칙**을 의미한다.

```
SOLID Principles

**S — Single Responsibility Principle**
**하나의 클래스나 함수는 하나의 책임만 가진다.**

O — Open/Closed Principle
기존 코드를 수정하지 않고 기능을 확장할 수 있어야 한다.

L — Liskov Substitution Principle
부모 타입을 자식 타입으로 대체해도 프로그램이 정상 동작해야 한다.

I — Interface Segregation Principle
하나의 거대한 인터페이스보다 여러 개의 작은 인터페이스로 분리한다.

D — Dependency Inversion Principle
구체적인 구현이 아니라 추상화에 의존하도록 설계한다.
```

---

## 2️⃣ 주석의 역설 - "How"가 아니라 "Why"

많은 개발자가 다음과 같은 주석을 작성한다.

```
for_user_in_users:        # 사용자 리스트를 순회
```

하지만 이런 주석은 코드와 동일한 정보를 반복할 뿐이다.

더 좋은 주석은 **설계 의도를 설명하는 것**이다.

예시

```
# 외부 API 호출 비용이 높기 때문에
# 먼저 캐시에 저장된 데이터를 확인한다
cached_data=cache.get(user_id)
```

이런 주석은 미래의 개발자가

**왜 이 코드를 이렇게 작성했는지 이해하도록 도와준다.**

---

## 3️⃣ 명확한 에러 메시지 작성

에러 메시지는 **개발자 경험(DX)** 에 큰 영향을 미친다.

나쁜 예시 : 원인을 알 수 없는 모호한 에러 메시지

```
raiseException("Error")     # 어떤 문제인지 알 수 없는 에러 메시지
```

좋은 예시 : 문제의 원인을 명확히 설명하는 에러 메시지

```
raiseValueError("Invalid user_id: user does not exist")
# 존재하지 않는 사용자 ID일 때 발생하는 에러
```

명확한 에러 메시지는

- 디버깅 시간 단축
- 운영 안정성 향상

으로 이어진다.

# 본론 3: Technical Writing - 코드 밖의 커뮤니케이션

앞서 말했듯이 **Clean Code가 코드 내부의 커뮤니케이션이라면**

**Technical Writing은 코드 외부의 커뮤니케이션이다.**

코드만으로 전달하기 어려운 **프로젝트의 구조, 사용 방법, 설계 의도 등을 문서로 설명하는 것**을 의미한다.

특히 협업 환경에서는 다음과 같은 상황이 자주 발생한다.

- 새로운 팀원이 프로젝트에 참여하는 경우
- 다른 팀이 우리의 API를 사용하는 경우
- 몇 달 후 내가 다시 코드를 보게 되는 경우

이때 문서가 없다면 **코드를 직접 읽어가며 구조를 파악해야 하는 문제**가 발생한다.

그래서 많은 프로젝트에서는 **README나 개발 문서 작성이 매우 중요한 역할**을 한다.

---

## 1️⃣ README 문서

가장 대표적인 Technical Writing의 예는 **README 문서**이다.

README는 보통 다음과 같은 구조를 가진다.

1️⃣ 프로젝트 개요

2️⃣ 기술 스택

3️⃣ 실행 방법

4️⃣ 프로젝트 구조

5️⃣ 핵심 로직 설명

예시 : 프로젝트의 디렉토리 구조를 README에 정리

```
Project
 ├ data     # 데이터 저장 및 전처리
 ├ model    # 머신러닝 모델 코드
 ├ api      # 서비스 API
 └ utils    # 공통 유틸리티 함수
```

이러한 구조 설명은 **프로젝트에 처음 참여하는 개발자가 전체 구조를 빠르게 이해하는 데 도움을 준다.**

---

## 2️⃣ API 문서

Technical Writing의 또 다른 중요한 예는 **API 문서**이다.

API 문서에는 보통 다음과 같은 정보가 포함된다.

- API endpoint
- 요청 방식 (GET / POST 등)
- 요청 파라미터
- 응답 데이터 형식

예를 들어 다음과 같이 작성할 수 있다.

```
GET /users/{user_id}

Description:
특정 사용자 정보를 조회한다.

Response:
{
  "user_id": 1,
  "name": "Alice",
  "email": "alice@example.com"
}
```

이런 문서가 있으면 **다른 개발자가 코드를 직접 읽지 않아도 API를 쉽게 사용할 수 있다.**

---

## 3️⃣ 시각적인 문서화

또한 복잡한 시스템에서는 **텍스트보다 시각적인 문서화가 더 효과적일 때도 많다.**

예를 들어

- 데이터 파이프라인
- 서비스 아키텍처
- API 호출 흐름

같은 구조는 코드나 글로 설명하는 것보다, 아래의 예시와 같이 **다이어그램으로 표현하는 것이 훨씬 이해하기 쉽다.**

- 데이터 흐름 다이어그램
- 시스템 아키텍처 다이어그램
- 서비스 구성도

이러한 문서는 **시스템 전체 구조를 한눈에 이해하도록 도와준다.**

---

# 5. 정리

Clean Code와 Technical Writing은 단순한 코드 스타일이 아니라

**협업 생산성을 높이는 핵심 개발 역량**이다.

핵심 원칙을 정리하면 다음과 같다.

좋은 코드의 특징

- 의미 있는 네이밍
- 작은 함수 구조
- 의도를 설명하는 주석

좋은 문서의 특징

- 명확한 README
- 구조화된 프로젝트 설명
- 시각적인 시스템 문서화

결론

> 좋은 개발자는 코드를 작성하는 사람이 아니라
> 
> 
> **다른 개발자가 이해할 수 있는 시스템을 만드는 사람**이다.
>
