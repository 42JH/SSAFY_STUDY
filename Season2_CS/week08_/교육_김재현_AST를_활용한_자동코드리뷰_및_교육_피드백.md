# 주제 : **AST를 활용한 자동 코드 리뷰 및 교육 피드백**

---

## 1. 문제 제기

### **왜 코드 리뷰는 어려운가?**

- 학생 수가 늘어날수록 코드 수도 증가
- 사람이 하나하나 리뷰하기에는 시간 부족
- 피드백 기준이 사람마다 달라질 수 있음

👉 핵심 문제

> “모든 학생에게 **일관되고 질 높은 피드백**을 제공할 수 있는가?”
> 

---

## 2. 해결 아이디어

### **코드를 ‘읽는 것’이 아니라 ‘분석’한다**

- 단순 문자열 비교가 아닌,
- 코드의 구조를 이해한다.

👉 질문

> “컴퓨터는 코드를 어떻게 이해할까?”
> 

---

## 3. 핵심 개념 ①

### **AST (Abstract Syntax Tree, 추상 구문 트리)**

👉 정의

> 코드를 **의미 중심의 트리 구조로 표현한 것**
> 

---

## 4. 🔍 AST 예시 (직관 이해)

코드:

```python
a = b + c
```

---

### 👤 사람의 입장

- b + c를 계산하고
- 그 결과를 a에 대입

---

### 💻 컴퓨터의 초기 상태

```
"a", " ", "=", " ", "b", " ", "+", " ", "c"
```

👉 단순한 문자들의 나열일 뿐임.

---

### 핵심 메시지

> 컴퓨터는 처음부터 의미를 이해하지 못하고,
> 
> 
> **해석 과정을 통해 의미를 만든다**
> 

---

## 그럼 어떻게 이해하게 될까?

👉 답: **중간 과정이 필요하다**

---

### 1️⃣ 토큰화 (Tokenizing)

문자를 의미 단위로 분리

```
[a] [=] [b] [+] [c]
```

---

### 2️⃣ 파싱 (Parsing)

코드의 구조를 파악

- “이건 대입문이다”
- “오른쪽은 덧셈 연산이다”

---

### 3️⃣ AST 생성

구조를 트리 형태로 표현

```
    =
   / \
  a   +
     / \
    b   c
```

---

## 직관적 비유

- 코드 = 문장
- AST = 문장 구조 분석

예시:

> “나는 밥을 먹는다”
> 
> 
> → 주어 / 동사 / 목적어로 나누는 것과 같다
> 

---

## 핵심 정리

- 코드는 처음엔 **텍스트**
- 분석 과정을 거쳐 **구조(AST)**로 변환
- 이 구조를 기반으로 **이해와 분석이 가능해진다**

---

## 4. 핵심 개념 ②

**정적 분석 (Static Analysis)**

정의:

> 코드를 실행하지 않고, 코드의 구조(AST)를 기반으로 규칙을 적용해 문제를 찾는 기술
> 

**정적 분석의 핵심 원리**

> AST를 순회하며 **미리 정의된 규칙(rule)**과 비교하여 문제를 탐지한다
> 

**정적 분석의 특징**

장점

- 빠름 (실행 필요 없음)
- 안전함 (코드 실행 안 함)
- 자동화 가능 (대규모 코드 처리 가능)

한계

- 실제 실행 결과는 알 수 없음
- 논리적 오류까지는 완벽히 못 잡음

👉 결론

> 따라서 **사람의 판단과 함께 사용하는 것이 가장 효과적이다.**
> 

---

## 5. 정적 분석으로 가능한 것

- 문법 오류 탐지
- 변수 사용 오류
- 비효율 코드 탐지
- 코드 스타일 검사

👉 **핵심**

> 실행 없이도 코드의 구조만으로 다양한 문제를 발견할 수 있다.
> 

---

## 6. 핵심 개념의 연결

**AST + 정적 분석 = 자동 코드 리뷰**

👉 동작 흐름:

1. 코드 → AST 변환
2. AST 탐색
3. 규칙 기반 패턴 탐지
4. 피드백 생성

---

## 7. 예시 ①

**불필요한 반복문 탐지**

코드:

```python
for i in range(len(arr)):      # 인덱스 i를 사용하여
		print(arr[i])              # arr의 i번째 인덱스에 해당하는 값을 출력
```

👉 AST 분석 결과:

- 인덱스 접근 패턴 발견

👉 문제점:

- 가독성이 떨어지고, 파이썬다운 코드(Pythonic)가 아니다.

💡 피드백:

> “for element in arr 사용 권장”
> 

```python
for element in arr:           # 인덱스를 사용하지 않고 바로 element로 선언
		print(element)            # element 출력
```

---

## 8. 예시 ②

**사용되지 않는 변수 탐지**

```python
x = 10            # 변수 선언
y = 20
print(x)          # x를 출력
```

👉 AST 분석:

- y 노드가 사용되지 않음

👉 문제점:

- 현재 사용되지 않는 변수(노드)가 존재함
- 코드 가독성과 유지보수성 저하
- 언어에 따라서 오류 발생 가능성

💡 피드백:

> “사용되지 않는 변수 제거”
> 

```python
x = 10            # 사용하지 않는 변수 삭제
print(x)
```

---

## 9. 예시 ③

**복잡도 문제 탐지**

- 중첩 반복문 구조 감지

👉 AST에서:

- 반복문 내부에 또 다른 반복문 존재 (Ex. for → for 구조)

👉 문제점:

- 입력 크기가 커질 경우 실행 시간이 급격히 증가할 가능성 존재
- 비효율적인 알고리즘 가능성

💡 피드백:

> “시간복잡도 O(n²) 가능성 있음”
>
